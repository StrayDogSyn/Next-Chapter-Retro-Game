# Session Log

Full chronological record of every AI-paired session on this project. The summary table in [AGENTIC_WORKFLOW.md](AGENTIC_WORKFLOW.md) shows the 3-5 most recent — this file is the complete archive.

**How to add an entry:** copy the template below, fill it in immediately after a session while it's fresh, don't batch these up.

---

## Template

```markdown
### YYYY-MM-DD — [Short session title]

- **Tool used:** [e.g. Copilot cloud agent, Claude, ChatGPT]
- **Goal:** What you asked the agent to do
- **Prompt summary:** One or two sentences (link to full prompt in PROMPT_LIBRARY.md if reusable)
- **What the agent produced:** Brief description
- **Human review/changes:** What you kept as-is, what you edited, what you rejected and why
- **Outcome:** ✅ merged / 🟡 partial / ❌ discarded
- **Time saved vs. hand-writing (rough estimate):**
- **Anything worth remembering:**
```

### Incident entry variant (for agent failures, false completions, verification catches)
Use heading format `### YYYY-MM-DD — [incident] <title>` with fields:
**What happened / Root cause of the failure / Resolution / Status of the underlying bug** (OPEN or FIXED-see-entry-dated-X).
An incident entry never doubles as the fix record — the fix gets its own dated entry with verification evidence.

```markdown
### YYYY-MM-DD — [incident] <title>

- **What happened:** What the agent did wrong and why it mattered.
- **Root cause of the failure:** The underlying reason (pattern-match, skipped audit, unverified claim, etc.).
- **Resolution:** What corrective action was taken (prompt rejection, redirect, re-run, etc.).
- **Status of the underlying bug:** OPEN — [brief description of what remains] / FIXED — see entry dated YYYY-MM-DD.
```

### 2026-07-14 — Fix Pack Increment 1: hero render scale corrected, S derived not assumed

- **Tool used:** Claude Code
- **Goal:** Increment 1 of the "Fix Pack: Hero Scale + Reachable Platforms" prompt — restore the swm hero's on-screen stature to roughly the retired `hero_0.png`'s, without touching the physics hitbox.
- **Audit findings (measured, contradicting the prompt's own framing):** `TILE = 16` (`lib/game/world.ts`). The prompt's context section assumed a uniform-scale bug ("expect S=2 for 16px-native art in a 32px world") and asked to derive S rather than assume it — doing so found that assumption was wrong. `drawW`/`drawH` in `drawPlayer()` were **32/34 both before and after** the ADR-020 hero swap (confirmed via `git show 27bbed4~1:lib/game/game.ts` — byte-identical values), so the destination render box never shrank. What changed: the retired `hero_0.png` was pipeline-packed via `pack_rows()`, which trims each frame to its content bbox and rescales it to fill its cell — measured fill ratio **37/48 ≈ 0.7708** on its `walkRight` row (4 frames, all exactly 37px tall). `char-sheet-alpha.png` was copied into the pipeline as-is (ADR-020 explicitly skipped `pack_rows()` since the source was "already alpha'd and grid-regular"), so its cells carry real unfilled padding — measured fill ratio **32/46 ≈ 0.6957** on its `run` row (median of 6 frames: `[32,25,33,32,33,32]`, median 32; the 25 outlier is a genuine shorter mid-stride pose, not a measurement error). Same destination box, less of it filled by opaque pixels — the character reads smaller without any literal "scale" ever being set below 1.
- **Live verification of the audit before implementing:** rather than trust the arithmetic alone, measured the actual on-screen character silhouette in a Playwright canvas screenshot (color-difference bounding box against the sampled background) before and after: pre-fix height 36 canvas-px, confirming a real (if more modest than the prompt's "~1 tile vs ~2 tiles" framing suggested) size deficit - not a 2x regression.
- **Decision:** Used the render pipeline's existing per-entity draw-scale mechanism (every sheet already takes its own `drawW`/`drawH` at the `drawSheetAnim()` call site, the same pattern `ENEMY_DEFS[...].drawW/drawH` uses) rather than regenerating the sheet at the pipeline level — satisfies the prompt's own decision rule ("if the render pipeline already supports a per-sheet or per-entity draw scale... set hero draw scale to S there") without touching pixel data. `HERO_SCALE = 0.7708/0.6957 ≈ 1.108`; new `drawW = round(32×1.108) = 35`, `drawH = round(34×1.108) = 38`. Anchor formula (`dx = px + pw/2 - drawW/2`, `dy = py + ph - drawH`) is unchanged — it already derives from `drawW`/`drawH` as variables, so it picked up the new size automatically; hitbox (`pw=14, ph=26`) untouched.
- **Live re-verification after the fix:** post-fix measured height 43 canvas-px (a 19% increase, in the same direction and rough magnitude as the derived 10.8% source-side correction plus normal cross-frame/measurement variance). Screenshots confirm: feet still on the ground line, no clipping, walking both directions still faces correctly (gun/body mirror cleanly left vs. right, matching ADR-020's native-facing+flip design — not disturbed by this change).
- **Outcome:** ✅ merged. `npm test` 60/60, `npm run build` exit 0, `npx tsc --noEmit` exit 0.
- **Anything worth remembering:** The prompt's own "derive, don't assume — paste the arithmetic" instruction caught its own framing being imprecise (S=2 would have been roughly double the actually-needed correction, and would have been applied via the wrong mechanism if pipeline-regeneration had been chosen reflexively instead of checking for an existing draw-scale path first). Third instance this project has now hit of a prompt's narrative assumption not surviving direct measurement (after F1/double-jump and ADR-020's grid/pose findings) — the fix each time was the same: measure before implementing, not after.
